# Packwright implementation plan — 2026-07-16

This file is the durable checklist for findings from the 0.1.0 install,
agent-operation, and migration probes. Update it in the same commit as each
completed work item so partially fixed contracts do not disappear from view.

## Status and rules

- `[x]` completed and verified
- `[-]` mitigated, with a remaining mechanical gap recorded below
- `[ ]` not started
- Work on one numbered work package at a time unless a release-blocking safety
  fix must interrupt it.
- Preserve deterministic, zero-network builds. Pin external integration
  versions; never resolve a floating `main` branch during install or migration.
- Keep the canonical mechanism semantic and runtime-neutral. Runtime names,
  paths, registration commands, and generated artifact lists belong in the
  adapter registry, built manifest, or migration receipt.
- Keep old mechanism specs readable for at least one compatibility cycle.

## Baseline

- PyPI release: `packwright==0.1.0`
- Current repository baseline: commit `049b7f2` (`Harden agent-guided setup and adoption`)
- The baseline changes below are committed but not yet released under a new
  package version.
- Baseline verification: 71 tests passed; quick release gate, zero-network
  audit, and public-tree audit passed.

## Planned order

1. Adapter-boundary foundation and old-spec normalization
2. Emotion Engine v1.0.0 state + MCP integration as one work package
3. Runtime-neutral multi-skill projection
4. English and Simplified Chinese compiler locale
5. Remaining workflow ergonomics and wording cleanup

Emotion Engine state migration and generic runtime installation intentionally
stay together. They share the upstream version, state contract, projected MCP
payload, manifest ownership, doctor/refresh behavior, and migration tests.

## WP1 — Adapter-boundary foundation

Goal: adding a new runtime should require a new adapter implementation, not
edits to every existing character or skill.

- [x] Introduce one adapter registry for entry paths, native skill/rule layout,
  imports, hooks/lifecycle projection, MCP registration, and capabilities.
- [x] Stop requiring `targets.mvp_adapter`, `targets.supported`, and
  `targets.reserved` in canonical character mechanisms.
- [x] Remove generated `outputs.<adapter>.artifacts` from canonical mechanisms;
  derive artifacts in the compiler and record them in manifest/receipt output.
- [x] Remove `emotion.projection.<adapter>` from the canonical mechanism. WP2
  owns the runtime-neutral activation, MCP transport, and state contract that
  replace it.
- [x] Replace `session_start.hook: SessionStart` with a semantic lifecycle event;
  adapters decide whether it becomes a hook, rule, or entry-file instruction.
- [x] Move runtime capability and ownership matrices out of character-owned
  source. Embed a resolved snapshot in the built pack when auditability needs it.
- [x] Remove `coverage` references to `outputs.codex`, `outputs.claude-code`, and
  other generated adapter paths.
- [x] Normalize 0.5/0.6 legacy specs into the new internal representation without
  rewriting user source during read, build, score, doctor, or migration.
- [x] Infer install adapter from the pack manifest. Keep `--adapter` only as an
  optional assertion/override and reject disagreement. This closes AD8.
- [x] Verify old-spec build and migration across Codex, Claude Code, and Cursor.

Acceptance:

- A fixture character with no runtime names in its canonical mechanism builds
  and scores on every registered adapter.
- A synthetic fourth adapter can register without editing that fixture.
- Existing 0.1.0 targets remain diagnosable and migratable.

Verification: 75 tests passed in the quick release gate; zero-network and
public-tree audits passed. Coverage includes a partial 0.5 adapter declaration,
an embedded legacy-spec migration, manifest-based install inference, all three
registered adapters, and a synthetic fourth adapter registry entry.

## WP2 — Emotion Engine v1.0.0 state and MCP integration

Goal: install one pinned, shared Emotion Engine implementation and let adapters
project only client-specific layout and registration.

- [x] Pin the current stable upstream release, Emotion Engine `v1.0.0`, and
  record its version plus source digest/commit in manifest and receipts.
- [x] Consume the upstream repository/package root, shared state schema, core
  helpers, stdio MCP server, and registration helper. Do not maintain a forked
  engine implementation inside each adapter.
- [x] Add adapter-neutral `--include-emotion-engine` and
  `--emotion-engine-source`; retain Codex-specific flags as deprecated aliases
  for one compatibility cycle.
- [x] Define `.emotion-engine/state.json` as the new project-local canonical
  path. Read and migrate `.emotion-engine/codex-state.json` and
  `.emotion-engine/emotion-state.json` as legacy candidates.
- [x] If multiple legacy state candidates exist and differ, stop and require an
  explicit choice. Never silently merge, reset, or overwrite live state.
- [x] Project the appropriate upstream skill/rule package plus MCP registration
  for each adapter. Adapter code owns paths and client configuration only.
- [x] Report Emotion Engine state as `active`, `snapshot_inert`, or `not_carried` in dry-run
  and apply receipts.
- [x] Generalize install, manifest bookkeeping, doctor, refresh, repair, and
  migration away from `emotion-engine-codex` names.
- [x] Preserve existing live state during install `--force`, refresh, same-runtime
  migration, and cross-runtime migration.
- [x] Keep install and migration zero-network; an unavailable or unsupported
  source version must fail with a concrete remediation message.
- [x] Add Codex, Claude Code, Cursor, legacy-state, conflict, MCP registration,
  doctor, refresh, and round-trip migration tests.

Acceptance:

- The same v1.0.0 state survives Codex -> Claude Code -> Cursor -> Codex without
  reset or schema drift.
- Each MCP-capable target points to the same explicit project-local state file.
- `doctor` detects payload/config drift without treating live state as a
  deterministic file eligible for replacement.

Verification: 79 tests passed in the quick release gate; zero-network and
public-tree audits passed. Coverage includes project MCP handshakes for all
three adapters, unrelated MCP-entry preservation, both legacy state paths,
conflicting-state zero-write refusal, force/refresh preservation, source-free
doctor diagnosis, and a byte-identical Codex -> Claude Code -> Cursor -> Codex
state round trip.

## WP3 — Runtime-neutral multi-skill projection

Goal: every semantic skill in the canonical mechanism becomes a native,
runtime-discoverable skill/rule without listing runtime names in the skill.

- [x] Treat entries under `skills:` as natively projected by default.
- [x] Keep reference-only material outside `skills:` instead of adding a
  `reference_only` adapter exception to each skill.
- [x] Allow optional capability requirements such as `mcp`, `hooks`, `shell`,
  `web`, or `browser`; do not add an `adapters:` list.
- [x] Let the adapter registry choose destination path, filename, front matter,
  import/routing syntax, and any supported degradation behavior.
- [x] Replace save-context-only output enumeration in generation, build,
  manifest, migration, doctor, and checker code with iteration over skills.
- [x] Keep skill semantics and source content platform-neutral; verify that
  adapter renderers do not inject runtime-specific meaning.
- [x] Distinguish Packwright-managed projected skills from carried unmanaged
  root `skills/` files in receipts and diagnostics.
- [x] Add a fixture with multiple skills and verify native projection plus
  cross-runtime migration on every registered adapter.

Acceptance:

- Adding a semantic skill changes only the canonical `skills:` list and source
  file; no runtime list or per-runtime artifact list is edited.
- Adding a new adapter changes no existing character or skill source.
- Checker and doctor validate all managed skills, not only save-context.

Verification: 83 tests passed in the quick release gate; zero-network and
public-tree audits passed. Coverage includes a second semantic skill projected
to all three native layouts, capability-based `hooks` degradation without an
`adapters:` field, entry routing, checker/manifest consistency, doctor repair,
and separately receipted unmanaged root-skill migration.

## WP4 — English and Simplified Chinese locale

Goal: localize compiler-owned boilerplate without translating user-authored
identity, memory, or skill content.

- [x] Support `en` and `zh-CN`; absent or unsupported locale falls back to `en`.
- [x] Do not attempt automatic support for other languages in this cycle.
- [x] Make the interviewer use English or Chinese when clearly established by
  the conversation, otherwise English. Remove the hard-coded Chinese A/B/C
  question from `intake_contract.py`. This closes AD7.
- [x] Centralize compiler-owned headings, fixed behavioral guidance, descriptions,
  and next-step messages in a locale catalog shared by all adapters.
- [x] Make checker rules locale-aware through semantic section identifiers or a
  resolved projection contract instead of isolated English string matching.
- [x] Keep canonical user-authored prose unchanged and continue using
  `allow_unicode` serialization.
- [x] Add full English and Chinese build/score/install/migrate fixtures for all
  registered adapters.

Acceptance:

- English remains the deterministic default.
- A `zh-CN` character produces consistently Chinese compiler boilerplate and a
  full checker pass without hand-editing generated artifacts.
- Unsupported-language input produces English boilerplate rather than a mixed
  or partially guessed localization.

Verification: `CharacterIntake.locale` resolves into `metadata.locale` and a
manifest locale projection contract. English and Chinese fixtures build, score
100.0, install, and migrate across Codex, Claude Code, and Cursor while keeping
user-authored role text byte-for-byte unchanged. Unknown locale input falls
back to English. The quick release gate passes 84 tests together with the
zero-network and public-tree audits.

## Completion protocol

For each work package:

1. Mark only the active item as in progress in the working notes or commit.
2. Add or update tests before marking an item complete.
3. Run the focused tests, full source suite, zero-network audit, public-tree
   audit, and the relevant multi-adapter release-gate paths.
4. Update this file in the implementation commit with concrete test/receipt
   evidence.
5. Do not mark a work package complete while a listed acceptance condition is
   unmet.
